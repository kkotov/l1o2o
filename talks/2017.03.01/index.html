Recap of L1T O2O in 2016

L1 trigger upgrade of 2015 rendered obsolete all of the system-specific O2O code

New code was gradually put in place through summer while running in the development mode (the run-critical uGT/L1Menu was prepared already in 2015 and always ran in production)

In the end of 2016 all (but CaloParams) payloads/tags were moved from Dev DB to Prod DB:

L1T O2O infrastructure

Job is invoked by SWATCH central cell in the end of configuration cycle and does the following:

  • with the given TSC key it navigates through subsystem tables and fetches the XMLs (online DB offers 3 tables per subsystem as explained in slide 5)
  • it parses the XMLs and packs offline-relevant parameters in payloads
  • it pushes the payloads to offline DB and stamps them with run-based IOV (slide 4)

One of the requirement is to start with pre-initialized payloads (CMSSW static configs or prototype payloads) and override a subset of parameters with values extracted from online DB

In a nutshell the L1T O2O is a CMSSW job wrapped in series of scripts (slide 5)

New L1T O2O features

O2O can be ran on lxplus provided you have the DB access token (great for testing!):

  • ./runL1-O2O-iov.sh 285832 collisions2016_TSC/v206 collisions2016_RS/v291

Splitting a monolithic job in a sequence per-payload jobs doubles running time (13s \(\rightarrow\) 25s)

  • for per-payload monitoring I'd prefer mimicking multiple jobs by splitting logs of one job

Tag name for any payloads can be changed with an argument (PR #17641 to be merged)

  • example: -t L1TCaloParams:Stage2v1_hlt,L1TMuonEndCap:TestTag instructs O2O to override the standard suffixes and produce L1TCaloParams_Stage2v1_hlt and L1TMuonEndCap_TestTag tags

Subsystem-specific workflows and comments

uGT: UTM XML-parsing library changes when new triggers are introduced in L1Menu

  • as agreed with AlCa, the library update comes only with a new central CMSSW installation

uGTrs: manually composed XML with errors crashed L1T O2O few times before

  • with the new L1CE service and new FM policy this problem is not expected to reoccur

CALO: eta-compress LUT is not in online DB \(\rightarrow\) prototype has to be kept-in-synch by experts

OMTF: no online configuration, new firmware on any change needed, but no changes in 2016

  • OMTF online software doesn't check FW version wrt. online DB \(\rightarrow\) O2O does nothing

EMTF: as in OMTF, firmware changes as opposed to online configs, except for 2GB pT LUT

  • online software checks LUT version wrt. online DB \(\rightarrow\) meta-LUTs (Forests) to be added ...

uGMT and BMTF are the most streamlined O2O components without special requirements
